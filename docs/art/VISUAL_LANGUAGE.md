# Purpose

This document defines the visual language of Project Genesis.

It specifies the visual building blocks used throughout the game to ensure that every asset shares a common design vocabulary.

The Visual Language bridges the gap between the high-level artistic vision and the detailed style guides for individual asset categories.

# Visual Identity

Project Genesis combines:

- modern business software
- industrial engineering
- city-building simulations
- technical illustrations
- contemporary information design

The visual language should always appear:

- clean
- structured
- precise
- timeless
- professional

---

# Geometry

## Preferred Shapes

Use primarily:

- rectangles
- rounded rectangles
- circles
- simple polygons
- hexagons for technology-related elements

Avoid:

- irregular shapes
- ornamental silhouettes
- exaggerated curves

Geometry should remain simple and purposeful.

---

# Corners

Rounded corners should communicate approachability without appearing playful.

Recommended corner radius categories:

| Usage | Radius |
|--------|--------|
| Buttons | Small |
| Panels | Medium |
| Dialogs | Medium |
| Cards | Medium |
| Icons | Minimal |
| Charts | None unless required |

Exact values will later be defined in DESIGN_TOKENS.md.

---

# Lines

Line work should be:

- consistent
- clean
- sharp
- minimal

Avoid:

- sketch effects
- variable-width strokes
- decorative outlines

---

# Borders

Borders should:

- separate information
- improve hierarchy
- remain subtle

Avoid thick borders.

Prefer soft contrast.

---

# Shadows

Shadows communicate elevation—not decoration.

Use:

- soft shadows
- low opacity
- consistent light direction

Avoid:

- dramatic shadows
- multiple shadow layers
- excessive blur

---

# Depth

Depth should be subtle.

Use layering to communicate:

- interaction
- focus
- modal dialogs
- floating elements

Avoid excessive three-dimensional effects.

---

# Lighting

Lighting should be:

- neutral
- evenly distributed
- realistic
- soft

Light direction should remain consistent across all assets.

Avoid cinematic lighting.

---

# Materials

Materials should appear authentic while remaining visually simplified.

Preferred materials:

- steel
- glass
- concrete
- brick
- wood
- vegetation
- asphalt
- water

Surface details should enhance recognition rather than realism.

---

# Surface Detail

Use medium detail.

Objects should remain recognizable even at reduced scale.

Avoid:

- excessive texture noise
- photorealistic micro details

---

# Perspective

Preferred perspective:

- Isometric

Alternative:

- Orthographic

Maps may use:

- Top-down projection

Perspective must remain consistent within each asset category.

---

# Scale

Objects should follow realistic proportions.

Buildings should appear believable relative to:

- roads
- vehicles
- people
- infrastructure

Visual exaggeration should only improve readability.

---

# Composition

Good composition should emphasize:

- balance
- symmetry where appropriate
- clear focal points
- whitespace
- predictable layouts

Avoid clutter.

---

# Color Usage

Color communicates meaning.

Primary purposes:

- hierarchy
- interaction
- category
- status

Decoration should never be the primary use of color.

Official values are defined in COLOR_PALETTE.md.

---

# Texture Philosophy

Textures should support recognition.

Textures should never dominate the asset.

Preferred:

- subtle
- clean
- realistic

Avoid:

- heavy grunge
- excessive wear
- visual noise

---

# Visual Density

Project Genesis favors medium visual density.

Each screen should feel information-rich without becoming overwhelming.

Spacing is an active design tool.

---

# Motion

Motion should communicate:

- state changes
- interaction
- progress
- completion

Animations should be:

- short
- subtle
- meaningful

Avoid unnecessary movement.

---

# Icon Language

Icons should be:

- simple
- geometric
- scalable
- immediately recognizable

Every icon should remain understandable at small sizes.

Detailed icon specifications are defined in ICON_GUIDELINES.md.

---

# Building Language

Buildings should express:

- purpose
- industry
- production chain
- technological level

Architecture should evolve naturally as technology advances.

---

# Map Language

Maps should prioritize information over decoration.

Layers should remain visually distinct.

Terrain should never obscure gameplay information.

---

# Dashboard Language

Dashboards should resemble modern business intelligence software.

Characteristics:

- structured
- spacious
- analytical
- data-focused

Players should quickly identify trends and anomalies.

---

# Visual Rhythm

The interface should establish a predictable rhythm through:

- consistent spacing
- repeated component sizes
- aligned elements
- balanced layouts

Visual rhythm improves usability and reduces cognitive load.

---

# Consistency Rules

Every new asset should maintain consistency in:

- geometry
- proportions
- lighting
- perspective
- color usage
- line weight
- shadow style
- spacing
- material rendering

Consistency is more important than stylistic experimentation.

---

# AI Asset Generation

When generating assets with AI, prompts should explicitly define:

- perspective
- geometry
- materials
- lighting
- level of detail
- color palette
- background
- intended use

Assets generated without these constraints should not become part of the official asset library.

# Summary

The Visual Language defines the shared visual vocabulary of Project Genesis.

By standardizing geometry, materials, lighting, composition and presentation, it ensures that every graphical asset contributes to a cohesive, professional and highly readable visual experience.
