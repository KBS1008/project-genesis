# Simulation Module

The `simulation` module contains the deterministic simulation engine of Project Genesis.

It is responsible for executing the game world in discrete simulation ticks.

The Simulation Layer coordinates execution but does not define business rules.

Business rules always remain inside the Domain Layer.

---

# Purpose

The simulation engine advances the game world by processing scheduled work in a deterministic and reproducible order.

Its primary goal is to guarantee that identical input always produces identical output.

---

# Responsibilities

The Simulation Layer is responsible for:

- Tick scheduling
- Simulation loop
- Time progression
- Event processing
- System execution
- Execution order
- Simulation state coordination

---

# Guiding Principles

The simulation must always be:

- deterministic
- reproducible
- predictable
- testable
- independent of the user interface
- independent of persistence

# Core Concepts

## Simulation Tick

The world advances in discrete ticks.

Each tick represents one logical simulation step.

The duration of a tick is configurable and must never be hardcoded.

---

## Simulation Engine

The Simulation Engine coordinates one complete simulation cycle.

Responsibilities include:

- advancing simulation time
- invoking systems
- processing events
- ensuring deterministic execution order

The engine should remain lightweight.

---

## Scheduler

The Scheduler determines which systems execute during each tick.

Responsibilities:

- register systems
- define execution order
- execute systems sequentially

Execution order must always be deterministic.

---

## Systems

A System performs one well-defined task during a simulation tick.

Examples:

- Production System
- Market System
- Finance System
- Transport System
- Research System

Systems should have a single responsibility.

---

## Simulation State

The Simulation Layer maintains execution state.

Examples:

- current tick
- simulation speed
- paused state
- running state

Business state belongs to the Domain Layer.

---

## Time

Simulation time is independent of real-world time.

Time progression should always be controlled through an abstract clock.

Never use:

```typescript
Date.now();
new Date();
```

inside simulation logic.

---

# Event Processing

The simulation processes Domain Events generated by Aggregates.

Typical flow:

```text
Aggregate

↓

Domain Event

↓

Event Queue

↓

Simulation Tick

↓

Event Dispatcher

↓

Subscribers
```

Events are processed in a deterministic order.

---

# Execution Order

A simulation tick should follow a consistent sequence.

Example:

```text
1. Advance Clock

2. Execute Scheduled Systems

3. Process Domain Events

4. Update Simulation State

5. Finish Tick
```

The exact sequence may evolve but must always remain deterministic.

---

# Dependency Rules

The Simulation Layer may depend on:

- Domain
- Common

It should use abstractions instead of concrete infrastructure implementations.

The Simulation Layer must not depend on:

- UI
- Infrastructure
- Content

---

# Error Handling

Expected simulation failures should be represented explicitly.

Preferred mechanisms:

- Result<T>
- SimulationError
- ValidationResult

Unexpected technical failures may use exceptions where appropriate.

---

# Performance

The simulation should prioritize:

- predictable execution
- low allocation rate
- stable execution time
- scalability

Premature optimization should be avoided.

---

# Testing

Recommended tests:

- Tick execution
- Scheduler order
- Event processing
- Time progression
- Deterministic replay
- Performance benchmarks

Determinism should be continuously verified through automated tests.

---

# Future Extensions

Possible future enhancements include:

- configurable tick rates
- replay support
- simulation snapshots
- multiplayer synchronization
- headless server mode

These extensions must preserve deterministic execution.
